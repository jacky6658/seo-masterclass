Chapter 1 : 技術 SEO 深度指南

1.1 robots.txt 完整語法

3 min

robots.txt 是放在網站根目錄的純文字檔，告訴爬蟲哪些可以爬、哪些不行。

User-agent: *
Allow: /
Disallow: /admin/
Disallow: /search?
Disallow: /tmp/

User-agent: Googlebot-Image
Disallow: /private-images/

Sitemap: https://example.com/sitemap.xml

常用指令

指令說明範例
User-agent指定適用的爬蟲User-agent: Googlebot
Disallow禁止爬取的路徑Disallow: /admin/
Allow允許爬取（覆蓋 Disallow）Allow: /admin/public/
Sitemap指向 sitemap 位置Sitemap: https://example.com/sitemap.xml
Crawl-delay請求間隔（Google 忽略此指令）Crawl-delay: 10
常見錯誤：Disallow: / 會封鎖整個網站。測試前永遠先用 GSC 的 robots.txt 測試工具驗證。

1.2 Sitemap 最佳實踐

3 min
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://example.com/</loc>
    <lastmod>2026-05-10</lastmod>
    <priority>1.0</priority>
  </url>
</urlset>

實戰重點

  • lastmod：只在內容真的更新時才改日期。亂改日期 Google 會忽略整個 sitemap 的 lastmod
  • priority：Google 官方已表示忽略此欄位。寫了無害但不要期待有效果
  • 上限：每個 sitemap 最多 50,000 個 URL 或 50MB。超過就用 sitemap index
  • 只放重要頁面：不要把 noindex 頁面、redirect 頁面、參數 URL 放進 sitemap

1.3 Canonical 進階用法

4 min
1

自參照 Canonical

每頁都指向自己的正版 URL。防止參數變體分散權重

2

跨頁 Canonical

內容重複的頁面指向主版本。例如印刷版指向原始版

3

跨域 Canonical

同步發佈到多個網域時，指定哪個是正版

4

動態頁 Canonical

有篩選/排序參數的頁面，canonical 指向無參數版本

注意：Canonical 是「建議」不是「指令」。如果 canonical 指向的頁面內容和當前頁差很多，Google 會忽略你的 canonical 自己決定。

1.4 URL 結構與 Redirect

3 min

URL 設計原則

好的壞的為什麼
/articles/tsmc-dividend//p?id=42語意化 URL 讓人和 Google 都看得懂
/tools/calculator//tools/calculator/index.html簡潔，避免多餘的檔名
3 層以內5+ 層巢狀扁平結構讓爬蟲更容易到達每頁
小寫 + 連字號大小寫混用 + 底線Google 把連字號當分隔，底線不是

301 vs 302 Redirect

301

永久轉址

URL 永久搬家。權重會傳遞。網域遷移、URL 結構改變時用

302

暫時轉址

URL 暫時搬家。權重不傳遞。A/B 測試、維護頁面時用

Ch.1 Quiz

robots.txt 的 Disallow: / 代表什麼？

A. 允許爬取所有頁面
B. 封鎖整個網站
C. 只封鎖首頁
Disallow: / 會封鎖根目錄下的所有路徑，等於封鎖整站。這是最常見的致命錯誤。

Google 會參考 sitemap 的哪個欄位？

A. loc 和 lastmod（如果日期準確）
B. priority 最重要
C. changefreq 決定爬取頻率
Google 只看 loc（URL）和 lastmod（最後更新日期），而且 lastmod 要準確才會被信任。priority 和 changefreq 已被忽略。

Chapter 2 : Core Web Vitals 優化

2.1 三大指標深度解析

4 min
LCP

Largest Contentful Paint

最大內容繪製時間
Good: < 2.5s
Poor: > 4.0s

INP

Interaction to Next Paint

互動反應延遲
Good: < 200ms
Poor: > 500ms

CLS

Cumulative Layout Shift

版面偏移量
Good: < 0.1
Poor: > 0.25

LCP 優化清單

  • 圖片壓縮：用 WebP/AVIF 格式，比 PNG 小 60-80%
  • Preload 關鍵資源：<link rel="preload" as="image" href="hero.webp">
  • CDN：Cloudflare Pages 已自帶。確認圖片也走 CDN
  • Lazy Loading：首屏以外的圖片加 loading="lazy"
  • 減少 CSS blocking：Critical CSS inline，其餘 async 載入

INP 優化清單

  • 拆分長任務：任何超過 50ms 的 JS 任務都要拆分
  • Defer 非關鍵 JS：GA、ads 腳本加 defer 或 async
  • 減少 DOM 大小：超過 1500 個節點就要注意

CLS 優化清單

  • 圖片加 width/height：瀏覽器預留空間，載入時不跳動
  • 字體：用 font-display: swap + preload 字體檔
  • 廣告預留空間：AdSense 區塊用 CSS 設定固定最小高度

2.2 測試工具差異

2 min
工具數據來源適合
PageSpeed InsightsCrUX 真實用戶 + Lighthouse 模擬最權威。Google 排名用的就是 CrUX 數據
Lighthouse (DevTools)本地模擬開發時即時測試。注意：你的電腦性能影響結果
GSC CWV ReportCrUX 真實用戶大盤趨勢。看整站有多少 URL Good/Poor
Web Vitals Extension你自己的瀏覽器即時看當前頁面的 CWV 數值
重要：PageSpeed Insights 分數不是排名因子。Google 用的是 CrUX 數據中的 Good/Needs Improvement/Poor 分類。分數 80 和 95 在排名上沒有差別，只要在 Good 範圍內就行。

Ch.2 Quiz

LCP 的「Good」標準是？

A. 低於 2.5 秒
B. 低於 1.0 秒
C. 低於 5.0 秒
LCP Good 標準是 2.5 秒以內。超過 4 秒是 Poor。

CLS 最常見的原因是？

A. JavaScript 太多
B. 圖片/廣告沒有預留空間
C. 伺服器太慢
CLS 主要是因為圖片、廣告、字體載入時把其他內容擠開。給圖片加 width/height、廣告區塊預留高度就能解決大部分問題。

Chapter 3 : 內容策略精通

3.1 搜尋意圖四分類

4 min
I

Informational

想知道答案
「殖利率是什麼」
給教學文章

N

Navigational

找特定網站
「Yahoo 股市」
你搶不到這種詞

C

Commercial

比較研究中
「0056 vs 00878」
給比較文章

T

Transactional

想採取行動
「股利計算器」
給工具/產品頁

怎麼判斷意圖：直接搜那個關鍵字，看 SERP 前 10 名是什麼類型的頁面。如果前 10 都是教學文，你就寫教學文；都是工具，你就做工具。不要硬用錯的內容類型去打。

3.2 Topic Cluster 完整建構

5 min
P
Pillar Page核心主題頁
廣度覆蓋
←→
C1
Cluster 1子主題深度
內部連結回 Pillar
←→
C2
Cluster 2另一子主題
也連回 Pillar

以你的股利計算器為例：

角色頁面關鍵字
Pillar首頁（股利計算器）股利計算器、殖利率計算
Cluster台積電配息計算台積電股利、2330 配息
Cluster0056 配息計算0056 股利、0056 殖利率

每篇 Cluster 文章在內文中連回 Pillar（首頁計算器），Pillar 也連到各 Cluster。這樣 Google 理解你對「台股股利」這整個主題有完整覆蓋。

A. Informational
B. Commercial
C. Transactional
使用者想找工具來用 = Transactional。所以你應該提供計算器工具頁，不是寫一篇「什麼是股利計算器」的文章。

Topic Cluster 的 Pillar Page 應該？

A. 廣度覆蓋核心主題，連結到各子頁
B. 深度寫一個子主題就好

Chapter 4 : 連結建設

4.1 Linkable Asset 策略

4 min

Linkable Asset = 別人會主動想連結到的內容。你不用求人連結，他們自己會做。

TOOL

免費工具

你的股利計算器就是最好的例子。投資部落客寫文時會自然引用「你可以用這個工具算」

DATA

原創數據報告

「2026 台股高殖利率排行榜」「金融股 10 年配息趨勢分析」。獨家數據人人想引用

INFO

資訊圖表

把複雜概念做成視覺化圖表。容易被轉貼分享

GUIDE

終極指南

一篇 3000+ 字的完整指南。成為某主題的首選參考資料

4.2 Guest Post 與 Digital PR

3 min

正確的 Guest Post vs 垃圾 Guest Post

正確做法垃圾做法
目標網站主題相關、有真實讀者任何願意收文的站
內容品質你會放在自己網站上的水準隨便湊字數
連結數量1-2 個自然的上下文連結塞滿連結
Anchor Text自然的品牌名或描述性文字精準匹配關鍵字
頻率每月 2-3 篇一天發 10 個站

Broken Link Building

找到相關網站上壞掉的連結（404），通知站長，建議用你的內容替代。步驟：

  1. 用 Ahrefs 找目標網站的 broken outbound links
  2. 確認你有對應主題的內容（沒有就寫一篇）
  3. 寫一封友善的 email 通知站長，附上你的替代連結
  4. 成功率約 5-15%，但得到的連結品質很高

Chapter 5 : Schema 結構化資料

5.1 JSON-LD 基礎與常用類型

5 min
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [{
    "@type": "Question",
    "name": "股利怎麼算？",
    "acceptedAnswer": {
      "@type": "Answer",
      "text": "總股利 = 持有股數 x 每股股利"
    }
  }]
}
</script>

常用 Schema 類型

類型用途Rich Result
FAQPage常見問題頁面搜尋結果展開 FAQ
HowTo步驟教學搜尋結果顯示步驟
Article文章/部落格新聞輪播、AMP
WebApplication線上工具應用程式資訊卡
Product產品頁價格、評分星星
LocalBusiness在地商家地圖、營業時間
BreadcrumbList麵包屑導航搜尋結果顯示路徑
Speakable語音搜尋告訴 AI 哪段適合朗讀
AEO 關鍵數據：82.5% 被 Google AI Overview 引用的頁面都有 Schema 結構化資料。有 Schema 被引用的機率是沒有的 2.5-3 倍。

5.2 巢狀 Schema 與測試

3 min

一頁可以放多個 Schema。你的首頁就是好例子：同時有 WebApplication + FAQPage + HowTo。

寫法：用 JSON 陣列包起來：

<script type="application/ld+json">
[{
  "@context": "https://schema.org",
  "@type": "WebApplication",
  "name": "股利計算器"
},
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [...]
}]
</script>

測試工具

RRT

Rich Results Test

Google 官方工具。測試你的 Schema 能不能觸發 Rich Results

SMV

Schema Markup Validator

schema.org 官方。檢查語法是否正確

Chapter 6 : 進階白帽技術

6.1 Internal Linking 與 PageRank 流動

4 min

內部連結不只是「讓使用者找到相關頁面」。它決定了 PageRank（權重）在你的網站內如何分配。

  • 首頁權重最高：因為首頁收到最多外部連結。從首頁連出去的頁面分到最多權重
  • Orphan Pages：沒有任何內部連結指向的頁面 = 權重孤島。Google 很難發現和排名它
  • Link Equity 分配：一頁連出去的連結越多，每條連結分到的權重越少。不要在頁尾塞 100 個連結
  • Contextual Link > Nav Link：文章內文中的連結比導航列的連結傳遞更多權重
實戰：這就是為什麼我們幫你在文章「內文中」加了連結，不只是底部的延伸閱讀。內文中自然出現的連結 Google 給予更高的權重傳遞。

6.2 Content Pruning 與 SEO A/B Testing

4 min

Content Pruning（內容修剪）

100 頁品質好 > 500 頁平庸。Helpful Content System 是 site-wide signal，低品質頁面會拖累好頁面。

頁面狀況處理方式
有流量但內容過時更新內容、保留 URL
主題重複的多頁合併成一頁完整版、舊 URL 301 redirect
零流量 + 零連結 + 低品質noindex 或直接刪除
零流量但有外部連結改善內容或 301 到相關頁面

SEO A/B Testing

不要猜，用數據驗證。最簡單的 A/B 測試：

  1. 選一批低 CTR 的頁面（GSC 看曝光高但點擊低的）
  2. 改寫 title tag（更吸引人、更清楚回答搜尋意圖）
  3. 等 2-4 週，比較 CTR 變化
  4. 有效就推廣到其他頁面，無效就繼續測試

6.3 Featured Snippet 優化

3 min

Featured Snippet = 搜尋結果頂端的特殊區塊（Position 0）。三種類型：

P

段落型

直接回答問題。在問題下方寫 40-60 字的簡潔答案

L

列表型

步驟或清單。用 H2/H3 + ol/ul 結構

T

表格型

比較數據。用 HTML table 呈現

搶 Snippet 的技巧：在 H2 用問句（「股利怎麼算？」），緊接著在下一段用 40-60 字直接回答。Google 很容易提取這種格式。你的 FAQ 段落就是完美的候選。

Ch.6 Quiz

內文中的連結 vs 導航列的連結，哪個傳遞更多 PageRank？

A. 內文中的連結（Contextual Link）
B. 導航列的連結
C. 兩者相同
Google 給內文中自然出現的連結更高的權重傳遞。導航列連結每頁都有，權重被稀釋。這就是在文章內文加連結比只放底部延伸閱讀更有效的原因。

Content Pruning 的核心原則是？

A. 刪除所有舊文章
B. 低品質頁面會拖累好頁面，質 > 量
C. 頁面越多排名越好
Helpful Content System 是 site-wide signal。整站有太多低品質頁面，好的頁面也會被拖下去。刪除或改善低品質頁面可以提升整站表現。
